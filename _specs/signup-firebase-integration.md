# Spec for signup-firebase-integration

branch: claude/feature/signup-firebase-integration

## Summary

Wire the existing signup form (`AuthForm` in `app/(public)/signup/`) to Firebase Authentication. On successful account creation, generate a random codename for the user by combining one word from each of three distinct word sets in PascalCase, set it as the user's Firebase `displayName`, and write a document to the `users` Firestore collection containing only the user's `id` and `codename` (no email). Use only the Firebase Web SDK via exports from `lib/firebase.ts`.

## Functional Requirements

- The signup form submits email and password to Firebase Auth using `createUserWithEmailAndPassword`.
- After account creation, a random codename is generated by selecting one word from each of three separate word lists and joining them in PascalCase (e.g. `SwiftSilentFox`).
- Each word list contains unique, thematic words. The three lists should feel cohesive and produce plausible heist-style codenames.
- The generated codename is set as the Firebase Auth user's `displayName` using `updateProfile`.
- A document is created in the Firestore `users` collection with the user's `uid` as the document ID, storing only `codename` and `id` fields â€” no email.
- The word lists and codename generation logic are extracted into a standalone utility function, separate from the form component.
- The form displays a loading/disabled state while the signup process is in progress.
- On success, the user is redirected to the dashboard (heists list page).
- On failure, a user-friendly error message is displayed in the form.

## Possible Edge Cases

- Firebase `createUserWithEmailAndPassword` may fail (e.g. email already in use, weak password) â€” the error must be caught and shown to the user without crashing.
- `updateProfile` or the Firestore `setDoc` call may fail after the auth account is created â€” handle gracefully without leaving the user in a broken state.
- The generated codename should not be empty â€” the utility must always return a valid string if the word lists are non-empty.
- Submitting the form multiple times quickly should be prevented while a request is in progress.

## Acceptance Criteria

- Submitting valid email and password creates a Firebase Auth account.
- The created user's `displayName` is set to a randomly generated PascalCase codename.
- A document exists in Firestore `users/{uid}` with `id` and `codename` fields only â€” no email stored.
- The codename is generated from three word lists; each list contributes exactly one word per codename.
- The form is disabled/loading during submission.
- A descriptive error message appears in the form when signup fails.
- On success, the user is redirected to the dashboard.
- No email is stored in Firestore.

## Open Questions

- Should we validate email format before allowing submission? Light validation
- Should we enforce minimum password length requirements? No.
- Should form inputs persist when navigating between login/signup? No.
- Do we need "Remember me" checkbox on login form? No.
- Should we include "Forgot password" link on login form? No.
- What page should the user land on after successful signup â€” `/heists` or another dashboard entry point?
- Should the codename be checked for uniqueness before being assigned, or is collision acceptable given the word list size?

## Testing Guidelines

Create a test file(s) in the `./tests` folder for the new feature, and create meaningful tests for the following cases, without going too heavy:

- The codename generator utility always returns a non-empty PascalCase string.
- The codename generator produces output in the format `WordWordWord` (three capitalised segments).
- The signup form calls `createUserWithEmailAndPassword` with the entered email and password on submit.
- The signup form shows an error message when Firebase returns an error.
- The signup form is disabled while submission is in progress.
